# 多模态附件支持与项目存档升级 - 详细实施计划书

## 0. 方案决策纪实 (Architecture Review & Decisions)

### 0.1 核心需求与背景演进
本项目旨在彻底解决聊天系统中“附件（特别是图片）”的上传、存储、上下文传递及持久化问题。在深度讨论中，我们决定摒弃 Base64 直接存储的方案（因其 33% 的体积膨胀及内存压力），转而建立**“全链路二进制闭环”**。附件在本地以原始二进制存储，仅在请求大模型 API 的瞬间进行临时转码，确保了系统的高性能与稳定性。

### 0.2 三层对象模型的权衡
为了支撑复杂的图文混排引用，我们确立了三层解耦架构：
- **Layer 1 (UI层/chat)**：面向前端渲染，通过索引引用图片。
- **Layer 2 (上下文层/ask)**：AI 的真实记忆载体。在此层级，附件 `attachments` 正式作为顶级属性存在，而 `content` 始终为带有标签的字符串，确保存储结构对所有模型通用。
- **Layer 3 (协议层/raw JSON)**：AI 返回的原始数据，是动态解析的源头。这种设计实现了存储、表现与传输的完美分离。

### 0.3 引用协议：为什么是 `[attachid:UUID]`？
我们选择对齐系统现有的 `appid` 引用模式。采用“标签化”字符串可以将图文顺序、引用位置以极轻量的方式持久化，且方便渲染引擎进行正则替换。这比复杂的 JSON 数组更易于管理，且对不支持 Vision 的模型完全透明，不会破坏对话上下文。

### 0.4 出站与入站的解耦逻辑
关于 `aiAsk.coffee` 的通用性，我们达成了关键共识：它作为工具库，不应感知具体的项目业务逻辑。
- **出站 (Request)**：`aiAsk` 仅定义标准的 `attachments` 属性规范。如果 `ask` 包含该属性，它会自动执行“标签 -> 多模态数组”的转换并发送给模型。
- **入站 (Response/Normalizer)**：针对 OpenAI SDK 回复始终为字符串的现状，我们在业务层 `ioApi_chat.js` 的 `onResponse` 钩子中实现了一个“归一化解析器”。它负责扫描 Markdown 图片并提取、转存为本地文件，再回填标签。这确保了无论底层模型（GPT-4o, Gemini, Kimi）的原生格式如何，系统内部的记忆始终是归一化的。

### 0.5 存档升级：引入 `adm-zip` 的必要性
原本的 `upload/` 目录是临时的，存档仅含 JSON 导致图片易丢失。我们将 `.owo` 升级为标准的 **ZIP 项目包**。
- **因果逻辑**：选择 `adm-zip` 而非系统 `tar`，是为了确保在 Windows、macOS 和 Linux 上行为 100% 一致。ZIP 格式也让存档具有极高的“用户可访问性”——用户只需改名即可直接查看图片和 JSON 记录。同时，通过文件头嗅探机制，系统实现了对旧版纯 JSON 存档的完美向下兼容。

## 1. 核心架构与数据流 (Core Architecture)
系统采用上述三层对象模型，确保存储、逻辑、传输与表现的深度解耦：

- **Layer 1 (UI层)**：`chat` 对象。
    - 存储：`comData.data.chatLists`。
    - 表现：渲染时通过 `chat.ask.attachments` 访问图片信息，不直接修改内容。
- **Layer 2 (上下文层)**：`ask` 对象。
    - 存储：`aiAsk.asks` 数组。
    - **核心规范**：新增顶级属性 `attachments`（数组）存储元数据。`content` 字段**强制保持为原始字符串**，严禁结构化（Array）存储。
- **Layer 3 (协议层)**：AI 响应的原始 `content` JSON 解析。

## 2. 附件标签协议 (Tagging Protocol)
为了兼容性并统一引用逻辑，对齐系统现有的 `appid` 引用模式：
- **标签格式**：`[attachid:UUID]`。
- **优势**：纯文本存储，对所有大模型（即使不支持 Vision 的模型）友好，不会破坏对话上下文的连续性。

## 3. 双向解析引擎逻辑

### 3.1 出站解析 (Ask -> AI Request)
- **实现位置**：`aiAsk.coffee` 的 `sendAsk` 方法映射循环。
- **判定条件**：检测到 `ask.attachments` 属性。
- **转换逻辑**：
    1. 使用正则 `/(\[attachid:[^\]]+\])/g` 切割 `ask.content` 字符串。
    2. 映射片段：普通文本映射为 `{ type: "text", text: "..." }`；标签映射为从 `attachments` 中读取路径并临时转码的 `{ type: "image_url", image_url: { url: "data:..." } }`。
- **通用性**：`aiAsk` 库仅定义 `attachments` 数据结构规范，不涉及项目具体的业务逻辑。

### 3.2 入站解析与归一化 (AI Reply -> Assistant Ask)
- **实现位置**：`ioApi_chat.js` 的 `onResponse` 钩子。
- **环境背景**：项目使用 OpenAI SDK，AI 回复的 `content` 必定被归一化为单一字符串。
- **核心逻辑 (Normalizer)**：
    1. **正则提取**：在 `onResponse` 接收到 `reply` 引用后，扫描其 `content` 字符串中的 Markdown 图片语法 `![...](...)` 或 AI 生成的 `[attachid:xxx]` 标签。
    2. **附件还原**：解析图片 URL 或 Base64 数据，调用文件系统持久化至项目的 `media/` 目录。
    3. **引用回填**：生成 `attachid` 并重构 `reply.content`（还原为带标签的字符串），同步更新 `reply.attachments`元数据。
- **最终产物**：AI 记忆中存储的始终是“降级”后的标签化内容。

## 4. 项目存档升级方案 (Persistence)
解决 `upload/` 目录临时性问题，实现附件与记录的强绑定：

- **打包格式**：升级 `.owo` 存档为 **标准 ZIP 压缩包**（改名后缀）。
- **选型库**：引入 **`adm-zip`**。纯 Node 实现，确保 Windows 与 macOS 环境行为高度一致，高性能处理二进制流。
- **包内结构**：
    - `project.json`：存储原有的 `comData` 等会话记录。
    - `media/`：项目专属的附件库。
- **便携性**：用户可直接将 `.owo` 改名为 `.zip` 手动查看 JSON 或提取图片。
- **平滑兼容**：`projectManager.js` 加载时探测文件头：
    - 若以 `{` 开头：识别为**旧版纯文本 JSON**。
    - 否则：识别为**新版项目包**并执行解压读取。

## 5. 组件改动清单
- **`aiAsk.coffee`**：
    - `sendAsk` 内部增加多模态 Array 组装逻辑。
- **`ioApi_chat.js`**：
    - `onResponse` 内部实现“入站归一化解析器”。
- **`projectManager.js`**：
    - 移除纯文本读写，实现基于 `adm-zip` 的封包/解包导出逻辑。
- **`format.js`**：
    - 增加基于 `[attachid:xxx]` 标签的 `<img>` 标签转换逻辑。
- **`ChatInputBar.js`**：
    - 增加回形针图标及本地文件选中->上传->插标流程。

## 6. 验证计划
1. **多模态链路验证**：发送带标图片，确认 AI 能识别并针对性回复。
2. **AI 生成还原测试**：模拟 AI 返回 Markdown 图片，确认系统能自动转存并显示。
3. **混合系统兼容性**：在 Windows 上创建的 `.owo` 项目包需在 Mac 上无缝打开及编辑。
4. **老版本平滑升级**：导入既有的 `.json` 存档，并在保存后自动变成 `.owo` ZIP 结构。
