# 项目调查报告 - 20260324

## 1. 任务背景
本次任务旨在实现全站 UI 的主题化重塑（计划版本 V17）。主要通过 `getColor` 钩子系统，支持欧式棕（0）、明亮黄（1）、经典蓝白（2）三套方案，严格遵守纯色名映射架构。

## 2. 调查范围
涉及的核心文件及路径如下：
- **核心逻辑**：
  - `www/src/view/common/colorObj.js` (色表定义)
  - `www/src/view/common/getColor.js` (主题路由引擎)
  - `www/src/view/common/commonData.js` (主题状态存储)
- **外壳组件**：
  - `www/src/view/common/noticeBox.js` (侧边栏/弹窗)
  - `www/src/view/common/TitleBar.js` (标题栏)
  - `www/src/view/common/nav.js` (导航栏)
- **聊天组件**：
  - `www/src/view/chat/ChatInputBar.js` (输入栏)
  - `www/src/view/chat/ChatItem.js` (消息项)
  - `www/src/view/chat/ChatList.js` (消息列表)
- **全局与设置**：
  - `www/src/view/css/Css.js` (动态 CSS 注入)
  - `www/src/view/setting/setting.js` (设置界面)

## 3. 编码风格与规范调查
- **框架**：基于 Mithril.js 的函数式组件。
- **色彩管理**：目前存在硬编码 HEX 值的情况，部分已使用 `getColor` 但逻辑不统一。
- **命名规范**：
  - 变量名/函数名：驼峰式（camelCase），如 `themeColor`, `handleConfirm`。
  - 组件名：PascalCase（如 `Box`, `Tag`）或 camelCase（如 `noticeBox`）。
  - 文件命名：多使用 camelCase（如 `commonData.js`），少数全小写（如 `nav.js`）。
- **模块化**：使用 ES6 `import`/`export` 语法。

## 4. 项目结构整理
目录：`www/src/view/`
- `common/`：存放通用组件及核心数据/逻辑。
- `chat/`：存放聊天相关组件。
- `css/`：存放全局 CSS 定义。
- `setting/`：存放设置界面。

## 5. 影响范围评估
本次修改涉及全站最基础的色彩系统及几乎所有主要视图组件。
- **技术风险**：
  - 遗漏硬编码导致主题不统一。
  - `getColor` 调用频繁，需确保性能。
  - 动态 CSS (`Css.js`) 的注入需确保在主题切换时实时更新。
- **系统建议**：
  - 需严格执行“纯色名映射”，严禁在组件层出现语义化色名。
