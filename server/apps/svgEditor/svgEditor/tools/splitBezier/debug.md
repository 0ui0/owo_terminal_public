我制作了一个svg编辑器，希望能够模拟adobe flash软件在矢量绘图上的特殊的行为
一切皆可以切割
在flash的软件里
线条和填充好像是分离的独立对象
线条和线条的相交处会被切割
线条和填充之前会被切割（不管这个填充有没有边线，可能是虚拟边线）
填充和填充之间也会被切割：
  如果两个填充颜色相同，会合并，包括边界的并集和填充色块的合并
  如果两个填充颜色不同且相交，会
    首先切割边界的线条
    保留顶部图形
    消除顶部和底部相交且被顶部遮挡图形的部分
    底部的图形边界被相交部分裁剪开
    （视觉上还是顶部和底部的两个图形，但是底部图形被裁剪了）
    【重要】需要考虑带包含的回形区域

为了实现上述功能，我设计了
线条 --- 可以使用墨水瓶工具上色，激活（如果隐藏），如果封闭，可以被颜料桶填充工具填充
纯填充 --- 可视化的svg填充色块，可以被颜料桶工具填充，可以被墨水瓶工具为其创建填充组
填充组 --- 逻辑信息，绑定了线条和纯填充的关系（js引用），更新填充组会刷新纯填充的显示，填充组的线条允许隐藏

平移规则：如果填充组的元素发生平移，会立即拆散（删除）填充组
切割规则：
1. 任何纯填充不允许在画布独立存在，如果检测到，会立即使用墨水瓶工具为其创建填充组
2. 线条直接会相互切割，相似的线条会被合并（避免无限切割和元素爆炸）
涉及填充的切割
3. 线条和填充之间会互相切割 
4. 填充和填充之间会互相切割

在 splitBezier/main_bak.coffee
 文件中，涉及填充组（fillGroup）的切割和合并逻辑主要集中在 splitPaper 方法内的 if element.type is "fillGroup" 这个分支里。

为了处理这种复杂的 Flash 风格的矢量破坏性（Destructive）相交，它内部定义了一个名为 fn 的异步核心处理函数。

它的切割逻辑基于一种**“打碎边框 -> 收集线段 -> 重新查找拓扑闭合 -> 重新用油漆桶灌色”**的机制，而不是纯粹的几何多边形布尔运算（如 Clipper 算法）。

我为你梳理了这套逻辑的核心运行步骤如下：

1. 触发比对的两种模式
代码对 fillGroup 发起了两次 fn 遍历：

模式一：用当前 fillGroup 的边缘线，去和画布上所有的游离线条（没有归属填充组的单线，例如刚画上去的线）比对。
模式二：用当前 fillGroup 的边缘线，去和其他 fillGroup2 的边缘线比对（处理两个色块碰撞的情况）。
2. 线段级相交与暴力撕裂（Line Subdivide）
在 fn 内部，它开启双重循环，拿着两组线（本组的线 element vs 对方的线 element2）两两碰撞。

它首先用外接矩形（isIntersectBoundingBox）做快速剔除，然后用精密计算（isIntersectLine）确认是否真有交叉点。
一旦发生交叉，拿到交点的时间参数 t1 和 t2，立刻调用 element.subdivideBezier 把这两根相交的线段在交点处齐根切断，变成多根更小的新线段。
做一次微操 alignPoints 强行把切割后的数学浮点数误差抹平吸附。
接着，用切断产生的新短线，去替换掉原来数据结构和图层里的长线（replaceElement）。
注意这段的暴力重置机制： 因为只要发生了切割，线段的总数和网格就变了，所以代码里直接暴力重置了迭代指针（比如 gIndex = elements.length），强迫循环倒退重来，直到整个区域内所有的线绝对不再有除端点外的交叉线段为止。
3. 构建拓扑线段池（收集 newRegion）
等到所有的线段都被彻底打碎成“原子形态”后，系统开始收集可以用来重新填色的“网格线框”（newRegion 数组）。 被拉进这个候选池的线段包括：

当前 fillGroup 所有被打碎的边缘线。
当前 fillGroup 内部的镂空区域的线（childRegions）。
如果要切它的是一根游离短线：通过判断端点是否在区域内（fillGroup.isPointInRegion），把落在色块肚子里的线全部吸纳进来。
如果是和另一个 fillGroup2 碰撞且发生了边缘相交：把隔壁那个色块的边缘线也全部一股脑合并进这个大池子里。
4. 拓扑面积搜索（findRegions）
收集齐满地被切碎的线段后，它把这团线段池全丢给了一个外挂引擎 searchRegions = findRegions(newRegion)。 这个引擎沿着所有的线段端点狂奔，把所有严丝合缝闭合起来的图形多边形区域全部找出来。

5. 防误判比对（haveRegion）
找到一堆新的闭合区域后，它并没有立刻下手。它会去跟目前画板上已有的所有 fillGroup 做引用和长度级别的深度对比：

确保找出来的新区域，是不是其实跟原本就在画布上的那个组一模一样？
如果一模一样（只发生了轻微碰撞没有实质几何改变），就当无事发生放弃切割。
6. 毁尸灭迹与重生（The Refill 阶段）
如果比对发现确实产生了新的拓扑结构（比如被线割成了两半，或者两个色块融合出了新区域）：

摧毁旧组： 把这涉及到的旧填充组的纯色视图 SVG 元素（pureFill）直接从画布上抹掉！并且调用 fillGroup.break() 彻底粉碎它们的数据绑定。
倒下新颜色（核心动作！）： 核心代码执行了 data.bucketTool.fill(null, fillGroup.prop.fill, newRegion, false)！
它拿着第一个被遍历到的组的颜色（fillGroup.prop.fill），把它当作油漆桶，重新倒入前面辛辛苦苦拼接出的全新线框池（newRegion）里，让油漆桶重新在这些线段围出来的迷宫里灌一次色。
💡 架构维度的总结与潜在雷区
从全局视野来看，这套逻辑高度还原了早期 Flash 的核心处理机制——在 Flash 中，当你画一条线穿过一个色块，Flash就是这样在后台把色块的两边线段打断，然后把你新画的线融合进去，变成两个独立的封闭区域。

但这套代码的潜在系统性风险在于第 6 步及其引发的多米诺骨牌效应：

颜色劫持（Color Hijacking）： 代码中 data.bucketTool.fill(...) 强行使用了 fillGroup.prop.fill（相交双方中主导方的值）。这意味着一旦两个颜色不同的组相交，重构填充时，另一方的颜色必然会被首要遍历方的颜色覆盖吃掉。
性能与连环崩盘点： 当画面有成百上千根曲线时，切断（Subdivide）任何一处导致指针重置重来，以及把海量曲线交给 findRegions 做重新追踪和再次调用 bucketTool，这相当于发生局部修改导致了全局物理引擎的“连通图重绘”，在复杂交叉和未闭合浮点误差下极易出现不可预期的死循环（或者导致颜色乱跑）。
